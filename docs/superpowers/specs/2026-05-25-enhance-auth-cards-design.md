# Enhance auth cards вҖ” signin + signup

**Status:** approved 2026-05-25

## Goal

Bring `/signin` and `/signup` up to "mature SaaS auth" feel via six targeted upgrades: cross-link footer, show/hide password toggle, icon-prefixed inputs, real-time validation, Terms & Conditions checkbox on signup, password strength meter on signup. Stay tightly scoped вҖ” no brand header, no social login, no recovery flow.

## Non-goals

- No brand header (logo + tagline) on the cards вҖ” user explicitly skipped this.
- No social login / SSO вҖ” no backend support.
- No "forgot password" / recovery flow вҖ” no backend support.
- No "remember me" вҖ” Convex Auth handles session persistence already.
- No split-pane / hero layout.
- No reusable `PasswordInput` / `IconInput` / `PasswordStrength` primitives вҖ” premature abstraction for two consumers. Inline in `signin.tsx` and `signup.tsx`.
- No T&C versioning, re-acceptance on signin, or per-user "last accepted version" tracking.
- No real Terms or Privacy content вҖ” placeholder stub routes only.

## PR plumbing

Build on a new branch `enhance-auth-cards` off **post-merge `main`** after PR #6 (theme) lands. Both PRs would touch `signin.tsx` and `signup.tsx`; sequencing avoids rebase friction. The design itself has no theme dependencies вҖ” tweakcn tokens are applied automatically once #6 is in.

## Architectural decisions

1. **Inline in `signin.tsx` and `signup.tsx`.** No new auth-form primitives. The two forms diverge enough (signup has 4 fields + T&C + strength meter; signin has 2 + cross-link) that a shared abstraction would be net-negative.
2. **Real-time validation pattern: `onBlur` then `onChange`.** No validation while typing on first interaction; once the field has been blurred once, errors update live as the user fixes them. Industry standard. Avoids "punishing" feel.
3. **Auth-level errors stay at bottom.** Field-level validation (email format, password length) is inline; server-returned errors (wrong password, email taken) surface above the submit button per the current pattern.
4. **Single-color password strength meter.** No green for "Kuat" вҖ” palette has no positive-feedback color. Width + label conveys strength; one fill color (`bg-foreground`) on a track (`bg-muted`).
5. **T&C links to stub routes.** `/terms` and `/privacy` render minimal "coming soon" placeholders. Real content is a content-writing task, not engineering.
6. **Icons on all 4 signup inputs.** Visual consistency overrides strict scope вҖ” User on Nama, Coffee on Nama kafe, Mail on Email, Lock on Password. lucide-react is already imported via the homepage logo.

## Validation rules

| Field | Rule | Error message (id-ID) |
|---|---|---|
| Email | required | "Email wajib diisi." |
| Email | RFC-lite format (`/^[^\s@]+@[^\s@]+\.[^\s@]+$/`) | "Format email tidak valid." |
| Password (signin) | required | "Password wajib diisi." |
| Password (signup) | required + min 8 chars | "Password minimal 8 karakter." |
| Nama Anda | required | "Nama wajib diisi." |
| Nama kafe | required + max 80 | "Nama kafe wajib diisi." / "Nama kafe maksimal 80 karakter." |
| T&C | must be checked | "Kamu perlu menyetujui Syarat Layanan." |

Submit is disabled while any field is invalid OR T&C is unchecked (signup).

## Password strength heuristic

4 buckets, computed inline (pure function in `signup.tsx`):

| Bucket | Condition | Bar fill | Label |
|---|---|---|---|
| 0 | empty | 0% | (no label) |
| 1 | <8 chars | 33% | "Lemah" |
| 2 | 8+ chars **OR** has 2+ of {lowercase, uppercase, digit, symbol} | 66% | "Sedang" |
| 3 | 12+ chars **AND** has 3+ of {lowercase, uppercase, digit, symbol} | 100% | "Kuat" |

Bar: `bg-muted` track, `bg-foreground` fill, height `h-1`, width transitions on `width` (200ms ease).

## Risks + accepted casualties

| Risk | Mitigation |
|---|---|
| T&C links point to "coming soon" stubs | Acceptable pre-launch; real content is a content task |
| Real-time validation can feel punishing if too eager | Mitigated by `onBlur`-then-`onChange` pattern |
| Password strength meter is advisory, not enforced | Convex Auth enforces вүҘ8 chars at the API; meter is UX hint |
| Show/hide toggle exposes password to shoulder-surfers | User-controlled; off by default |
| Single-color strength bar conveys less than colored bars would | Accepted per strict palette rule |
| Form taller on signup | Still fits above 800px viewport; verified in visual sweep |
